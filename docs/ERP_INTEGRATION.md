# ERP 연동 설계 — 더존 ERP-iU / 옴니이솔

> 목적: PMS에서 저장한 데이터(수주·기성/대금·구매발주·원가 등)를 더존 ERP-iU와
> 양방향으로 연동(입력·수정·삭제)한다. 이 문서는 **무엇을 협의·설정하고, 어떤 행위를
> 시스템이 해야 하는지**를 정리하고, PMS에 추가된 연동 골격의 사용법을 설명한다.

---

## 1. 핵심 결론 (먼저 읽기)

1. **더존 ERP-iU는 공개 REST API로 자유롭게 붙는 구조가 아니다.** 실무 연동은 보통
   ① **자동전표/인터페이스 테이블 연동**, ② **연동 에이전트(Agent)**, ③ **옴니이솔(SI)이
   구축하는 커스텀 인터페이스(웹서비스/배치/DB)** 중 하나로 진행된다.
2. 따라서 **옴니이솔과의 인터페이스 규격 합의가 1순위 작업**이다. PMS는 그 규격이
   무엇이 되든 붙을 수 있도록 **어댑터(adapter) 구조 + 변경 아웃박스(outbox)**로
   준비해 두었다(이 코드).
3. 권장 구도: **PMS가 단일 진실원본(SoR)으로 데이터를 저장 → 변경분을 outbox에 적재 →
   동기화 워커가 옴니이솔이 제공한 인터페이스로 전송 → ERP가 전표/문서 생성 → 문서번호를
   PMS에 회신**. ERP가 원본인 마스터(거래처·품목·계정)는 ERP→PMS 단방향으로 받는다.

---

## 2. 더존 ERP-iU 연동 방식 비교

| 방식 | 설명 | 장점 | 단점 | PMS 적합도 |
|------|------|------|------|-----------|
| **인터페이스(스테이징) 테이블** | ERP DB에 약속된 연동 테이블을 두고 PMS가 그 테이블에 적재 → ERP 배치가 전표 생성 | 더존 표준에 가깝고 안정적 | ERP DB 접근/온프레미스 망 필요, 옴니이솔 설정 필수 | ★★★ (권장) |
| **연동 에이전트(Agent)** | 사내에 설치된 에이전트가 ERP 표준 API/DB와 PMS 사이를 중계 | 방화벽 안에서 동작, 검증된 패턴(비즈플레이 I-U Agent 등) | 에이전트 설치·운영, 옴니이솔/더존 협조 | ★★★ |
| **커스텀 웹서비스/REST** | 옴니이솔이 ERP 위에 REST/SOAP 엔드포인트를 구축 | PMS(클라우드)에서 바로 호출, 양방향 용이 | 별도 개발·비용, 옴니이솔 SI 범위 | ★★★ (클라우드 PMS엔 최적) |
| **Excel/CSV 배치 업로드** | 정해진 양식으로 내려받아 ERP에 일괄 등록 | 가장 단순·저비용, 즉시 시작 가능 | 수작업/지연, 수정·삭제 동기화 약함, 실시간 아님 | ★★ (1단계 또는 폴백) |

> PMS가 **Vercel 클라우드**에서 동작하므로, ERP가 온프레미스(사내)면 클라우드→사내 직접
> DB 접근은 보안상 막혀 있다. 이 경우 **에이전트** 또는 **옴니이솔의 외부 공개 REST 게이트웨이**가
> 필요하다. 망 구성이 핵심 협의 사항이다.

## 3. 옴니이솔/더존과 반드시 합의할 항목 (협의 체크리스트)

연동 착수 전 옴니이솔(또는 더존 기술팀)과 아래를 문서로 확정해야 한다.

**A. 인터페이스 방식·망**
- 연동 방식: 인터페이스 테이블 / 에이전트 / REST 중 무엇인가
- ERP 설치 형태: 온프레미스 vs 클라우드(더존 클라우드) — 망 접근 경로(VPN/전용선/에이전트)
- 인증: 계정/토큰/인증서, IP 허용목록

**B. 연동 대상 문서(전표) 매핑** — PMS의 어떤 저장이 ERP의 어떤 문서가 되는가
| PMS 데이터 | ERP-iU 대상(예시) |
|------------|-------------------|
| 수주(project) | 영업관리 · 수주등록 |
| 기성/대금 청구(billing, status=청구/확정) | 매출관리 · (세금)계산서 / 매출전표 |
| 대금 수금(billing status=지급/paid) | 자금관리 · 수금 전표 |
| 구매 발주(procurement, status=발주) | 구매관리 · 발주서 |
| 기자재 입고(procurement status=입고) | 구매관리 · 입고/매입 |
| 원가 집행(cost_entries) | 원가관리 · 원가전표 |
> 실제 모듈·전표 종류·계정과목은 회사 회계정책에 따라 옴니이솔과 확정.

**C. 마스터 코드 매핑** — 양 시스템의 코드 체계를 맞춘다
- 거래처(고객/발주처/공급사) 코드 ↔ PMS subcontractors/client
- 품목(기자재) 코드 ↔ procurement_items
- 계정과목/부서/프로젝트(공사) 코드 ↔ projects, work_packages
- 누가 마스터의 원본인가(보통 ERP가 원본) → PMS는 받아서 매핑

**D. 방향·키·멱등성**
- 방향: 트랜잭션(수주·기성·구매)은 PMS→ERP, 마스터(거래처·품목·계정)는 ERP→PMS 권장
- 키 매핑: PMS UUID ↔ ERP 문서번호(전표번호) — 1:1 매핑 보관
- 멱등성: 같은 변경을 두 번 보내도 전표가 중복 생성되지 않도록 **외부참조키(external_ref)** 사용
- 수정/삭제: ERP가 확정(승인)된 전표의 수정·삭제를 허용하는가? (보통 미승인만 가능)
  → PMS의 수정/삭제가 ERP에서 어떻게 처리되는지(역분개/취소전표 등) 규칙 합의

**E. 운영**
- 동기화 주기: 실시간(이벤트) vs 배치(분/시간 단위)
- 마감 처리: 회계 마감 이후 기간 데이터는 잠금
- 오류/재처리: 실패 건 재시도 정책, 담당자 알림, 수동 보정 화면
- 정합성 점검: 일/월 단위 대사(reconciliation)

## 4. PMS 측 설정·행위 (이 코드가 구현한 것)

PMS는 어떤 인터페이스가 와도 붙도록 **아웃박스 + 어댑터** 구조로 준비됐다.

### 4.1 저장 시 (요청 #1: "이 프로그램에서 저장 수행")
- 사용자가 PMS에서 대금/구매/수주를 저장(입력·수정·삭제)하면, **DB 트리거가 변경분을
  `erp_sync_outbox` 테이블에 자동 적재**한다(`migrations/0009`). AI 명령으로 바뀐 것도 동일하게 잡힌다.
- 즉 "저장"은 평소처럼 하면 되고, 연동 큐 적재는 자동이다.

### 4.2 ERP로 전송 (요청 #2: 입력/수정/삭제 연동)
- `/api/erp/sync` 가 outbox의 미처리 건을 읽어 **ERP 어댑터**로 전송한다.
  - 어댑터(`lib/erp/adapter.ts`)는 `create/update/delete(entity, payload)` 인터페이스.
  - 구현체: `mock`(개발), `staging`(인터페이스 테이블 적재), `rest`(옴니이솔 REST 호출) — 환경변수 `ERP_ADAPTER`로 선택.
- 성공 시 ERP가 회신한 문서번호를 outbox와 원본 레코드(`erp_doc_no`)에 기록 → 멱등성 보장.
- 동기화는 **Vercel Cron** 또는 수동 트리거로 주기 실행(보안 토큰으로 보호).

### 4.3 ERP→PMS 반영
- 마스터 변경/전표 상태 변경을 ERP(또는 옴니이솔 미들웨어)가 `/api/erp/webhook` 으로 통지 →
  PMS가 매핑을 갱신한다(서명 검증). 폴링 방식이면 별도 워커로 ERP를 읽어 반영.

### 4.4 설정(환경변수)
```
ERP_ADAPTER=mock            # mock | staging | rest
ERP_BASE_URL=               # rest 방식일 때 옴니이솔 엔드포인트
ERP_API_KEY=                # rest 인증 토큰
ERP_WEBHOOK_SECRET=         # ERP→PMS 인바운드 서명 검증
ERP_SYNC_SECRET=            # /api/erp/sync 호출 보호(크론용)
```

## 5. 데이터 흐름 (요약도)

```
[PMS 저장: 대금/구매/수주]
      │ (DB 트리거)
      ▼
 erp_sync_outbox  ── /api/erp/sync (Cron/수동) ──► ERP Adapter ──► 더존 ERP-iU
   (op, payload,                                   (staging/rest)     │ 전표 생성
    status, attempts)                                                  │
      ▲                                                                ▼
      └────────── erp_doc_no 회신·매핑 갱신 ◄───────────────────────────┘

[ERP 마스터/상태 변경] ── /api/erp/webhook ──► PMS 매핑·상태 반영
```

## 6. 단계별 도입 로드맵 (권장)

1. **0단계 — 합의**: 옴니이솔과 §3 체크리스트 확정(방식·망·매핑·규칙). *가장 중요.*
2. **1단계 — 단방향 배치**: 수주·기성·발주를 PMS→ERP로 1일 1회(또는 Excel)로 보내 검증.
   (이 코드의 `staging`/`rest` 어댑터 + outbox로 바로 가능)
3. **2단계 — 준실시간**: outbox를 분 단위 Cron으로 처리, 문서번호 회신·멱등성 확립.
4. **3단계 — 양방향**: 마스터 동기화 + 전표 상태 webhook 반영, 수정/삭제 규칙 적용.
5. **4단계 — 운영화**: 오류 재처리 화면, 일/월 대사 리포트, 마감 잠금.

---

## 7. 리스크·주의

- **회계 정합성**: 확정/승인된 전표는 임의 수정·삭제 불가가 일반적 → PMS의 수정/삭제를
  ERP에서 역분개/취소로 처리하는 규칙을 반드시 합의(데이터 깨짐 방지).
- **중복 전표**: 멱등성 키(external_ref) 없이 재전송하면 전표가 중복 생성됨 → outbox/매핑 필수.
- **망·보안**: 온프레미스 ERP면 클라우드 직접 DB 접근 금지 → 에이전트/게이트웨이 필요.
- **코드 불일치**: 거래처·품목·계정 코드가 양쪽에서 다르면 전송 실패 → 매핑 테이블 선구축.
- **마감**: 마감기간 데이터 전송 차단 로직 필요.

## 부록. 이 PMS에 추가된 연동 구성요소

- `supabase/migrations/0009_erp_integration.sql` — `erp_sync_outbox`, `erp_mapping`,
  원본 테이블의 `erp_doc_no`/`erp_synced_at` 컬럼, 저장 시 자동 적재 트리거, RLS.
- `lib/erp/adapter.ts` — `ErpAdapter` 인터페이스 + `mock`/`staging`/`rest` 구현 스켈레톤.
- `lib/erp/sync.ts` — outbox 처리 루프.
- `app/api/erp/sync/route.ts` — 아웃박스 전송 엔드포인트(토큰 보호, Cron 연결).
- `app/api/erp/webhook/route.ts` — ERP→PMS 인바운드(서명 검증).

> 실제 전송 로직(전표 필드 구성, 인증)은 옴니이솔 인터페이스 규격 확정 후
> `staging`/`rest` 어댑터의 TODO 부분을 채우면 된다. 구조·큐·멱등성·매핑은 이미 준비돼 있다.
